# 모니터링 연결 문제 분석 보고서

## 1. 개요
현재 `monitoring/` 디렉터리의 웹 대시보드가 `server/`의 C 서버와 통신하지 못하는 문제를 분석한 결과, 프로토콜 및 아키텍처 측면에서 두 가지 주요 결함이 발견되었습니다.

## 2. 주요 문제점

### 2.1 프로토콜 불일치 (WebSocket vs Raw TCP)
- **현상**: 브라우저의 `app.js`는 `ws://localhost:9000`으로 연결을 시도하지만, 서버는 응답하지 않거나 연결이 즉시 끊어짐.
- **원인**: 
    - `app.js`는 **WebSocket** 프로토콜을 사용합니다. WebSocket은 HTTP 핸드쉐이크와 특수한 데이터 프레임 구조를 가집니다.
    - `monitor_server.c`는 일반적인 **Raw TCP** 소켓 서버입니다. 브라우저의 WebSocket 요청(HTTP Upgrade)을 이해하지 못하고 무시하거나 잘못된 데이터를 보냅니다.
- **결과**: 브라우저 표준 보안 정책상 Raw TCP 서버에 직접 연결하는 것은 불가능합니다.

### 2.2 서버 루프 블로킹 (Blocking I/O)
- **현상**: 서버가 실행 중임에도 불구하고 새로운 모니터링 클라이언트가 접속되지 않음.
- **원인**:
    - `run_server.c`의 `run_client_loop` 함수 내에서 `recv_data`가 **블로킹(Blocking)** 방식으로 동작합니다.
    - 즉, 장비(Equipment)가 데이터를 보내지 않고 대기 중일 때 서버는 `recv` 함수에서 멈춰 있게 되며, 그 뒤에 있는 `monitor_accept_clients()` 함수가 호출되지 못합니다.
- **결과**: 장비가 데이터를 전송하는 찰나의 순간이 아니면 새로운 모니터링 접속이 허용되지 않습니다.

## 3. 해결 방안

### 3.1 Node.js 브리지(Bridge) 도입
- 브라우저(WebSocket)와 C 서버(TCP) 사이에서 중계 역할을 하는 `bridge.js`를 작성합니다.
- 브리지는 9000번 포트에서 WebSocket을 열고, C 서버의 9001번 포트(TCP)에 접속하여 데이터를 실시간으로 전달합니다.

### 3.2 C 서버 포트 변경 및 비차단(Non-blocking) 처리
- 모니터링 서버의 포트를 브리지가 접속할 9001번으로 변경합니다.
- 장비 데이터 수신과 모니터링 클라이언트 수락이 서로 방해받지 않도록 루프 구조를 개선합니다.

### 3.3 안정성 개선
- 장비 연결이 끊어져도 서버 프로세스가 종료되지 않고 다음 연결을 대기하도록 수정합니다.
